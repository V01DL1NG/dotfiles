# Dock Configuration TUI ŌĆö Design Spec

**Date:** 2026-03-20
**Status:** Approved

---

## Problem

On a new Mac the Dock contains Apple defaults. Every time a new machine is set up,
apps must be manually pinned and reordered. There is no way to reproduce a known
Dock layout automatically. The existing `role.sh` system switches shell environment
but leaves the Dock unchanged.

---

## Goals

- Interactive TUI to choose which apps appear in the Dock, per role (work/personal)
- Config persists in the repo as plain text files ŌĆö editable by hand, re-applicable non-interactively
- `role.sh apply work` automatically applies the saved work Dock layout
- Follows the `tmux-config.sh` TUI pattern (fzf multi-select + `select` variant choices)
- `--dry-run` support throughout
- Idempotent: safe to re-run

---

## Non-Goals

- Linux support (macOS only ŌĆö Dock is macOS-specific)
- Per-machine overrides (one config per role, same on every Mac)
- Automatic app installation (only apps already in `/Applications` or `~/Applications` are offered)

---

## Config Files

Two plain-text files committed to the repo:

```
dock/work.txt
dock/personal.txt
```

Format ŌĆö one entry per line:
- App path: `/Applications/Arc.app`
- Spacer: `---`
- Comment (ignored): lines starting with `#`
- Blank lines ignored

The token `---` is parsed by checking that the entire trimmed line equals `---`.
This is unambiguous and grep-safe.

Example `dock/work.txt`:
```
# Work Dock
/Applications/Arc.app
/Applications/Cursor.app
/Applications/iTerm.app
---
/Applications/Slack.app
/Applications/Zoom.app
---
/Applications/Finder.app
```

Files are generated by the TUI and committed to the repo. They can also be edited
by hand without running the TUI.

**Variant metadata** is stored as comment lines at the top of the config file so
`--apply` can reproduce the same layout non-interactively:

```
# dock-position: left
# dock-clear: yes
/Applications/Arc.app
```

`--apply` reads `dock-position` and `dock-clear` from these lines if present.
Defaults when absent: `position=bottom`, `clear=yes`.

**Default starter configs** (shipped in repo, overwritten by TUI on first run):

`dock/work.txt`:
```
/Applications/Finder.app
/System/Applications/Mail.app
/Applications/Arc.app
---
/Applications/Slack.app
```

`dock/personal.txt`:
```
/Applications/Finder.app
/Applications/Arc.app
/Applications/Safari.app
---
/System/Applications/Music.app
```

## Script: `dock-config.sh`

### Modes

```
./dock-config.sh                  # interactive TUI
./dock-config.sh --apply work     # silently apply saved work config
./dock-config.sh --apply personal # silently apply saved personal config
./dock-config.sh --dry-run        # show commands, no changes
```

`DOCK_CONFIG_DIR` env var overrides the default `$SCRIPT_DIR/dock/` path ŌĆö used in
tests to point the script at a fixture directory without touching the real configs.

### Interactive TUI Flow

**Entry point** ŌĆö Y/n prompt (same pattern as tmux-config.sh):
```
Customise Dock layout? [Y/n]
```

**Stage 1 ŌĆö Role selection** ŌĆö `select` menu:
```
Which role to configure?
  1) Work
  2) Personal
  3) Both
```
Selecting "Both" runs Stages 2ŌĆō3 twice in sequence (work first, then personal).

**Stage 2 ŌĆö App picker** ŌĆö fzf multi-select (fallback: numbered list if fzf not found):
- Scans `/Applications` and `~/Applications` for installed `.app` bundles
- Adds a special `--- SPACER ---` entry at the top of the list
- User searches and space-selects apps in desired order; select `--- SPACER ---`
  wherever a visual spacer is wanted in the Dock
- No pre-selection ŌĆö starts empty
- Prompt: `Select apps for <Role> Dock (space to select, Enter to confirm)`

The spacer entry is the mechanism for placing `---` lines in the config: wherever
the user selected `--- SPACER ---`, a `---` line is written in the config file.
There is no separate "Yes/No spacers" variant step ŌĆö spacer placement is fully
explicit via the picker.

**fzf ordering note:** fzf `--multi` returns items in input-list order, not
selection order. The TUI therefore cannot guarantee app ordering matches the
selection sequence. This is a known fzf limitation. The TUI is the mechanism for
picking apps; final ordering should be done by hand-editing the generated config
file. This is documented in the post-TUI output and in HANDBOOK.md.

**Stage 3 ŌĆö Variant choices** ŌĆö `select` loops:

1. Dock position (sets via `defaults write`, not dockutil ŌĆö dockutil has no position flag):
   ```bash
   defaults write com.apple.dock orientation -string "bottom"  # or "left" / "right"
   ```
   Options: `Bottom (default)` / `Left` / `Right`

2. Clear existing Dock before applying: `Yes (default)` / `No`

**Saves config** to `dock/<role>.txt` then immediately applies.

### `--apply <role>` Flow (non-interactive)

1. Resolve config file: `$DOCK_CONFIG_DIR/<role>.txt` (default: `$SCRIPT_DIR/dock/<role>.txt`)
2. If config file not found ŌåÆ `warn "No Dock config for role '<role>' ŌĆö skipping"` and `exit 0`
3. Install `dockutil` via Homebrew if not present
4. `dockutil --remove all --no-restart` ŌĆö clear current Dock
5. For each line in config:
   - Skip blank lines and lines starting with `#`
   - If trimmed line equals `---`: `dockutil --add '' --type spacer --section apps --no-restart`
   - Otherwise: `dockutil --add "/path/to/App.app" --no-restart`
   - If app path does not exist: `warn "Skipping missing app: /path/to/App.app"` and continue
6. `killall Dock` ŌĆö single restart after all items are added

**`--no-restart` must be passed on every `dockutil` call** (both remove and add).
A single `killall Dock` at the end is the only Dock restart. Omitting `--no-restart`
on any add call causes the Dock to visually flash and restart per item.

**Concrete dockutil command examples:**
```bash
dockutil --remove all --no-restart
dockutil --add /Applications/Arc.app --no-restart
dockutil --add '' --type spacer --section apps --no-restart
killall Dock
```

### `--dry-run`

Suppresses ALL writes and system calls ŌĆö both config file writes AND dockutil/killall
calls. Prints the commands that would run instead. Works for both interactive TUI
and `--apply` modes. Mirrors the `tmux-config.sh` pattern: `--dry-run` means
nothing touches the filesystem or system state.

### Dependency: `dockutil`

Added to `Brewfile`. Script checks for `dockutil` before any Dock operations and
installs via `brew install dockutil` if missing.

## `role.sh` Integration

The Dock apply call is placed **before the `is_applied` guard** in `cmd_apply`.
This ensures the Dock is always reconfigured when `role.sh apply` is called,
even if the zshrc role marker already exists (e.g. after restoring `.zshrc`
from backup on a reinstall).

Current `cmd_apply` structure (`role.sh:67-94`):
```bash
cmd_apply() {
  validate_role "$role"
  if is_applied "$role"; then   # ŌåÉ guard exits here if already applied
    warn "...already applied..."
    exit 0
  fi
  # zshrc write...
  success "Role applied."
}
```

The Dock call is inserted **before** the `is_applied` guard so it always runs:
```bash
cmd_apply() {
  validate_role "$role"

  # Always apply Dock ŌĆö runs even if zshrc role is already present
  if [ -f "$SCRIPT_DIR/dock-config.sh" ]; then
    bash "$SCRIPT_DIR/dock-config.sh" --apply "$role"
  fi

  if is_applied "$role"; then
    warn "...already applied..."
    exit 0
  fi
  # zshrc write...
}
```

## Testing: `test-dock-config.sh`

Follows the `test-touchid-sudo.sh` pattern. macOS-only (exits 0 with skip on Linux).

Uses `DOCK_CONFIG_DIR` env var to point the script at a temp fixture directory,
avoiding any reads or writes to the real `dock/` configs:

```bash
FIXTURE_DIR="$(mktemp -d)"
echo "/Applications/Finder.app" > "$FIXTURE_DIR/work.txt"
echo "---"                       >> "$FIXTURE_DIR/work.txt"
echo "/Applications/Arc.app"    >> "$FIXTURE_DIR/work.txt"
```

**Test sections:**

1. **Syntax** ŌĆö `bash -n dock-config.sh`

2. **`--dry-run` output** (using fixture via `DOCK_CONFIG_DIR`):
   - `dockutil --remove all` appears
   - `/Applications/Finder.app` appears in output
   - spacer command (`--type spacer`) appears (from `---` line in fixture)
   - `killall Dock` appears

3. **Arg parsing**:
   - `--apply work` with missing config (empty `DOCK_CONFIG_DIR`) exits 0
   - Invalid args exit non-zero
   - `--dry-run` exits 0

**`test.sh` wiring:**
- `syntax_check dock-config.sh` goes in the existing "Linux Platform ŌĆö Syntax Checks"
  block (alongside other scripts ŌĆö `bash -n` is platform-agnostic)
- The `test-dock-config.sh` section is macOS-guarded (same pattern as
  `test-touchid-sudo.sh`):
  ```bash
  if [ "$(uname -s)" != "Darwin" ]; then
    warn "test-dock-config.sh skipped (macOS only)"
  elif bash "$SCRIPT_DIR/test-dock-config.sh" >/dev/null 2>&1; then
    pass "test-dock-config.sh passed"
  else
    fail "test-dock-config.sh ŌĆö one or more checks failed"
  fi
  ```

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| `dockutil` not installed | Auto-install via brew before any Dock operations |
| `dock/<role>.txt` missing at role apply time | Exit 0 with warning ŌĆö role switch never blocked |
| App path in config no longer exists | Warn and skip ŌĆö continue applying rest of list |
| `--no-restart` omitted on an add call | Spec explicitly requires it on every dockutil call |
| Dock cleared but apply fails mid-way | `--dry-run` for preview; re-run to restore |
| fzf not available | Fallback to numbered `select` menu (same as tmux-config.sh) |
